# Arsitektur & Integrasi AI Berbasis SKILL sebagai Referensi

**Tools devops security** yang diusulkan memanfaatkan konsep _AI Agent Skills_ (folder berisi instruksi dan metadata) untuk memberi referensi yang relevan ke model AI saat melakukan pemindaian keamanan kode. _Agent Skills_ adalah direktori yang berisi file `SKILL.md` dengan frontmatter (nama, deskripsi, tag, dll) dan instruksi rinci【25†L350-L354】. Dengan cara ini, hanya _skill_ yang relevan yang dimuat oleh LLM (Large Language Model) sesuai kebutuhan, sehingga kontekstual dan efisien. Gambar di bawah menjelaskan arsitektur konsep AI dengan Agent, SKILL, dan lingkungan eksekusi (VM) yang saling terhubung, di mana SKILL disimpan di direktori tersendiri dan di-_load_ oleh agen AI sesuai tugas:

【69†embed_image】 _Contoh arsitektur agent AI dengan integrasi SKILL; LLM memuat SKILL dari direktori ketika perlu【25†L350-L354】._

Menurut DigitalOcean, SKILL memisahkan resource ke modul-modul tersendiri sehingga LLM hanya memuat instruksi tambahan saat relevan【25†L350-L354】. Agan-agin AI (seperti Claude Code, OpenAI Codex) diharapkan mampu menemukan, memuat, dan mengeksekusi skill dengan mencocokkan permintaan pengguna ke metadata skill yang sesuai【25†L350-L354】【25†L455-L462】. Protocol seperti _Model Context Protocol_ (MCP) turut mendukung pola ini dengan mendefinisikan konsep _Tools_, _Resources_, dan _Prompts_ untuk komunikasi antara model dan sumber eksternal【51†L129-L133】. Dengan SKILL sebagai referensi, pengetahuan domain (mis. daftar pola kerentanan, instruksi khusus bahasa) bisa ditambahkan ke LLM tanpa menyertakan semuanya dalam prompt. Namun perlu diperhatikan SKILL eksternal bersumber dari komunitas yang belum diverifikasi, sehingga potensi **injeksi perintah** berbahaya ada【51†L142-L150】【49†L142-L150】. Oleh karena itu, sistem harus memfilter atau memvalidasi SKILL (mis. digital signing) sebelum dipakai agar instruksi tidak membajak proses analisis.

## Ekstraksi Kode & Analisis AST

Untuk memeriksa kerentanan dalam kode sumber, tool harus mengurai (parse) kode menjadi struktur sintaks (AST). Di Golang, pustaka **`go/parser`** dan **`go/ast`** menyediakan cara built-in untuk membangun AST dari kode Go【56†L122-L125】. Paket ini memudahkan penelusuran pohon sintaks, misalnya menggunakan fungsi `ast.Inspect`, sehingga developer dapat menjalankan analisis statis atau mengekstrak informasi kode sebagai alat bantu testing/QA【56†L122-L125】. Misalnya, bila tool memeriksa Go, dapat langsung gunakan `parser.ParseFile` untuk menghasilkan AST, lalu menavigasi node untuk menemukan pola-pola bahaya (contoh: cek fungsi panggilan yang berbahaya).

Untuk mendukung **bahasa lain** (JavaScript, Python, dll.), salah satu solusi populer adalah **Tree-Sitter**. Tree-Sitter adalah pustaka parsing yang mampu menghasilkan AST untuk beragam bahasa pemrograman【21†L52-L60】. Ada binding Go (mis. `github.com/tree-sitter/go-tree-sitter`) dan grammar khusus per bahasa. Dengan Tree-Sitter kita bisa men-setup _parser_ dalam Go untuk bahasa tertentu, lalu memproses kode menjadi AST dan menelusurinya mirip contoh di Go【21†L52-L60】. Misalnya, contoh di [21] menunjukkan cara sederhana parsing JS di Go: membentuk `parser := ts.NewParser(); parser.SetLanguage(tree_sitter_javascript.Language()); tree := parser.Parse(code, nil)`, lalu ekstrak AST. Umumnya hasil AST bisa dibaca dengan format S-expression atau JSON, lalu diproses untuk menemukan pola kerentanan.

Selain itu, ada alat seperti **Semgrep** yang juga memanfaatkan AST (walau lebih ke pola matching) lintas bahasa. Namun Semgrep berbasis rule YAML, sementara kita bisa implementasi kustom dengan AST parser dan pola sendiri. Bagian pemeriksaan (deteksi) nantinya dapat mengombinasikan analisis statis sederhana (pola AST, regex) dan LLM untuk pemahaman semantik.

## Alur Pengecekan, Reasoning AI, & Mekanisme Loop

**Flow pemindaian** tipikal: tool akan mengambil setiap file kode sumber (berdasarkan bahasa yang didukung), parse menjadi AST, kemudian memeriksa AST tersebut terhadap daftar pola kerentanan tertentu. Setiap pola (mis. _buffer overflow_, _sql injection_, dll) bisa didefinisikan sebagai SKILL dengan contoh kode rentan atau instruksi LLM. Misalnya, SKILL bisa berisi _tag_ atau _regex_ yang menggambarkan celah XSS, dan LLM diarahkan meninjau AST terhadap pola tersebut. Alat ini dapat menggunakan _multi-engine detection_ seperti yang digunakan Cisco Skill Scanner: gabungan analisis statis berbasis pola (seperti YARA/regex), alur data (taint analysis), dan LLM sebagai “hakim” untuk konteks lebih dalam【59†L342-L345】. Pendekatan ini meningkatkan cakupan deteksi kerentanan sambil menekan false positive.

Dalam proses scan, tool bisa melakukan **deep reasoning** bertahap: misalnya dengan menerapkan _chain-of-thought prompting_ pada LLM, di mana model menganalisis satu file atau blok kode secara step-by-step. Skema loop umumnya: baca kode (AST) ➔ hitung metadata SKILL (tag/pattern) yang cocok ➔ kirim prompt ke LLM dengan instruksi dari SKILL terkait ➔ LLM mengeluarkan potensi temuan dan rekomendasi. LLM dapat diminta “mikir mendalam” (rumus _deep reasoning_) untuk merinci temuan, bahkan menyarankan fix. Pada akhirnya, hasil disimpan (mis. SARIF/JSON). Loop ini diulang file-per-file atau modul-per-modul. Selama loop juga sistem dapat melewatkan (skip) file/ekstensi tertentu sesuai konfigurasi.

**Mekanisme pengecualian (exclude)**: Tool harus bisa mengabaikan file/folder tertentu (misalnya binary, vendor, test, atau file yang tidak relevan). Contohnya, `gosec` (static analyzer Go) memiliki opsi `-exclude-dir` atau `-exclude-rules` untuk menyaring path atau aturan tertentu【66†L667-L670】. Kita dapat meniru konsep ini: misalnya konfigurasi berupa list glob (e.g. `*.min.js`, `node_modules/**`, `migrations/*`) atau filter folder (`/docs/, /build/`). File dengan ekstensi tertentu juga bisa dikecualikan (mis. `.md`, `.png`). SKILL atau rule parsing bisa menyertakan tag atau pola khusus sehingga loop SCAN dapat memutuskan “lewatkan file ini”. Dengan begitu scanning fokus pada kode sumber utama saja.

Selain itu, proses loop dapat mengedepankan optimasi: misalnya mengenali tipe file dulu (ekstensi) dan memilih AST parser yang tepat. Untuk file lintas bahasa, panggil parser sesuai. Setelah AST didapat, cek quick-fails (mis. jika frontmatter SKILL tidak cocok tag), baru lanjut prompt LLM. Jangan lupa juga deteksi _infinite loop_: batasi panjang prompt/respon LLM atau langkah reasoning agar tidak menggantung.

## Integrasi CI/CD: GitHub Actions, GitLab Runner, & MCP

Karena alat ini ditujukan untuk pipeline DevOps, harus mudah dipasang di CI/CD. Untuk **GitHub Actions**, kita dapat membuat _action_ (container atau JS) yang menjalankan binary Go. Misalnya, mirip `gosec` yang sudah tersedia sebagai Actions resmi【33†L525-L532】. Pipeline YAML dapat menggunakan langkah checkout + run tool kita dengan `uses: nama-org/namatool@vX`. Output scan bisa di-_upload_ sebagai file SARIF supaya GitHub Code Scanning menampilkan temuan di UI【33†L612-L620】.

Untuk **GitLab Runner**, caranya serupa: buat job di `.gitlab-ci.yml` yang memanggil container atau script tool. GitLab SAST mendukung output SARIF/JSON khusus agar hasil masuk ke _Merge Request Security Widget_ dan pipeline security page【43†L380-L389】. GitLab bahkan menyediakan “Custom Scanner Template” – cukup sertakan include template tersebut, maka CI/CD akan menjalankan scanner dan mengimpor hasilnya sesuai schema (vulnerability, secret, etc)【43†L380-L389】. Artinya kita harus output JSON sesuai spec GitLab SAST (misalnya _Generic SAST report_ atau _Secret Detection_ schema) agar hasil terdaftar di dasbor GitLab【41†L348-L354】【43†L380-L389】.

Untuk **Model Context Protocol (MCP)**, protokol ini memungkinkan komponen eksternal (Tools, Resources) dipanggil oleh agen AI. Kita bisa membungkus tool sebagai “MCP Tool” sehingga agen (LLM) dapat memanggilnya via stdio/HTTP【51†L129-L133】【69†】. Misalnya, agen memberi perintah `scan /path/to/code`, tool membaca input dari STDIN, melakukan scan, lalu mengembalikan output JSON ke agen. Dengan MCP, tool ini bisa diintegrasikan dalam arsitektur agentic AI sebagai _Resources_ yang dijalankan on-demand. Dokumen [69†] menunjukkan contohnya: agen AI memanggil SKILL/tool di dalam VM sesuai kebutuhan【69†】.

## Implementasi Golang, Pengujian, & Keamanan Runtime

Karena dipilih Golang, tool ini akan mudah dikompilasi ke binary lintas platform. Struktur rekomendasi: gunakan modul Go, pisahkan komponen _parsing_ (pakai `go/parser` atau _go-tree-sitter_) dan _analisis_ (termasuk komponen pemanggilan LLM). Manfaatkan concurrency Go: misalnya proses file parallel goroutine untuk efisiensi. Sertakan opsi command-line (flags) untuk konfigurasi direktori SKILL, folder kerentanan, batasan file, dsb.

Pengujian dapat mencakup: **unit test** (mis. fungsi parser AST mengembalikan struktur node yang diharapkan), **integration test** dengan contoh kode rentan, dan **CI test** (mis. jalankan scan sample project di GitHub Actions/GitLab CI untuk hasil SARIF sesuai). Pastikan antar-langsung LLM (jika ada) diuji dengan kunci dummy saat CI.

Keamanan runtime perlu diperhatikan: meski tool melakukan static scan, ia akan memproses kode yang mungkin berbahaya. Hindari _exec_ atau menjalankan kode korban. Gunakan AST murni (tidak exec) untuk mencegah remote code execution. Jika memakai LLM eksternal, lindungi API key dengan variable environment, dan hindari mengekspos rahasia di log. Pertimbangkan validasi SKILL yang di-_load_: SKILL eksternal harus diasumsikan untrusted【49†L142-L150】. Misalnya, terapkan whitelist atau sandbox processing (MCP resources contohnya).

Terakhir, karena tool ini aman tapi kompleks, terapkan _best practices_ Golang: error handling yang rapi, sanitasi input, pembaruan dependency, dan pantau isu CVE di pustaka (misalnya modul Go AST jarang rawan). Dengan manajemen memory/goroutine yang benar, tool dapat berjalan ringan di runner CI dan MCP.
